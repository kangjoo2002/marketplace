# A-1a Products Single-Column Index Attempt Observations

## Scope

This document records observations from the single-column index attempt artifacts in `results/`.

The observations are limited to actual EXPLAIN output generated by `products_single_column_index_attempts.sql`. These are experiment indexes, not accepted production indexes.

## status Index Attempt

In the generated artifacts, the planner did not choose `idx_exp_<profile>_status` for Q1-Q6 on any of the three profiles.

| Q case | observed access pattern | sort behavior | artifact observation |
|---|---|---|---|
| Q1 | `Parallel Seq Scan` on all profiles | `Gather Merge`; `top-N heapsort` | `status` appears as a filter, but the single-column status index was not used. |
| Q2 | `Parallel Seq Scan` on all profiles | `top-N heapsort`; `Gather` or `Gather Merge` depending on profile | The narrower category/brand/status/price shape still did not use the status index. |
| Q3 | `Parallel Seq Scan` on all profiles | `Gather Merge`; `top-N heapsort` | The review-count sort case did not use the status index. |
| Q4 | `Parallel Seq Scan` on all profiles | `Gather Merge`; `quicksort` or `top-N heapsort` depending on profile | Deep OFFSET did not make the status index the chosen access path. |
| Q5 | `Parallel Seq Scan` on all profiles | `Gather Merge`; `top-N heapsort` | Even the status-only query used parallel sequential scan in these artifacts. |
| Q6 | `Parallel Seq Scan` on all profiles | `Gather Merge`; `top-N heapsort` | Q6 does not filter by status, so the status index was not relevant. |

What this suggests: in these local artifacts, `status` alone was not selective enough, or not useful enough for the requested ordering, for PostgreSQL to choose it.

What not to conclude yet: do not claim that all status indexes are useless. This only covers one single-column experiment index and this synthetic workload.

## price Index Attempt

The planner chose `idx_exp_<profile>_price` for Q6 on all three profiles. It did not choose the price index for Q1-Q5 in the generated artifacts.

| Q case | observed access pattern | sort behavior | artifact observation |
|---|---|---|---|
| Q1 | `Parallel Seq Scan` on all profiles | `Gather Merge`; `top-N heapsort` | The core category/status/price/recent sort query did not use the price index. |
| Q2 | `Parallel Seq Scan` on all profiles | `top-N heapsort`; `Gather` or `Gather Merge` depending on profile | Even with `price ASC`, the category/brand/status filters prevented the price-only index from becoming the chosen plan. |
| Q3 | `Parallel Seq Scan` on all profiles | `Gather Merge`; `top-N heapsort` | Q3 does not filter by price and did not use the price index. |
| Q4 | `Parallel Seq Scan` on all profiles | `Gather Merge`; `quicksort` or `top-N heapsort` depending on profile | Deep OFFSET with category/status/price/recent sort did not use the price index. |
| Q5 | `Parallel Seq Scan` on all profiles | `Gather Merge`; `top-N heapsort` | Q5 does not filter by price and did not use the price index. |
| Q6 | `Index Scan using idx_exp_<profile>_price` on all profiles | `Incremental Sort` | Q6 is the price-only control case. Execution Time was `2.459 ms` on uniform, `40.095 ms` on moderate skew, and `1.108 ms` on high skew. |

What this suggests: price-only indexing matched Q6's broad price-range and `price ASC, id ASC` shape, but it did not solve the core multi-condition Q1/Q2 shapes in these artifacts.

What not to conclude yet: do not interpret Q6 behavior as proof that a price index solves the full products search workload.

## created_at Index Attempt

The planner chose `idx_exp_<profile>_created_at` for Q1 and Q5 on all three profiles. It chose the created_at index for Q4 on moderate skew and high skew, but not on uniform. It did not choose the created_at index for Q2, Q3, or Q6.

| Q case | observed access pattern | sort behavior | artifact observation |
|---|---|---|---|
| Q1 | `Index Scan Backward` or `Parallel Index Scan Backward` using the created_at index | `Incremental Sort` | The index matched the recent-first direction, but category/status/price filters still appeared as filters. |
| Q2 | `Parallel Seq Scan` on all profiles | `top-N heapsort`; `Gather` or `Gather Merge` depending on profile | Q2 sorts by price, not created_at. |
| Q3 | `Parallel Seq Scan` on all profiles | `Gather Merge`; `top-N heapsort` | Q3 sorts by review_count, not created_at. |
| Q4 | uniform: `Parallel Seq Scan`; moderate/high: `Parallel Index Scan Backward` using created_at | uniform: `quicksort`; moderate/high: `Incremental Sort` | Deep OFFSET with created_at index was profile-dependent in these artifacts. Moderate skew recorded Execution Time `5859.528 ms`; high skew recorded `1510.228 ms`. |
| Q5 | `Index Scan Backward using idx_exp_<profile>_created_at` on all profiles | `Incremental Sort` | Q5 is broad status-only with recent-first ordering. Execution Time was `1.091 ms` on uniform, `0.882 ms` on moderate skew, and `1.423 ms` on high skew. |
| Q6 | `Parallel Seq Scan` on all profiles | `Gather Merge`; `top-N heapsort` | Q6 sorts by price, not created_at. |

What this suggests: `created_at` can be attractive to the planner for recent-first cases, but created_at alone does not encode category, status, or price selectivity.

What not to conclude yet: do not treat the Q5 access path as proof that created_at alone is the correct index for Q1/Q4 or the overall products search workload.

## review_count Index Attempt

The planner chose `idx_exp_<profile>_review_count` for Q3 on all three profiles. It did not choose the review_count index for Q1, Q2, Q4, Q5, or Q6.

| Q case | observed access pattern | sort behavior | artifact observation |
|---|---|---|---|
| Q1 | `Parallel Seq Scan` on all profiles | `Gather Merge`; `top-N heapsort` | Q1 sorts by created_at, not review_count. |
| Q2 | `Parallel Seq Scan` on all profiles | `top-N heapsort`; `Gather` or `Gather Merge` depending on profile | Q2 sorts by price, not review_count. |
| Q3 | `Index Scan Backward using idx_exp_<profile>_review_count` on all profiles | `Incremental Sort` | Q3 is the review-count sort case. Execution Time was `312.095 ms` on uniform, `196.966 ms` on moderate skew, and `55.302 ms` on high skew. |
| Q4 | `Parallel Seq Scan` on all profiles | `Gather Merge`; `quicksort` or `top-N heapsort` depending on profile | Q4 sorts by created_at, not review_count. |
| Q5 | `Parallel Seq Scan` on all profiles | `Gather Merge`; `top-N heapsort` | Q5 sorts by created_at, not review_count. |
| Q6 | `Parallel Seq Scan` on all profiles | `Gather Merge`; `top-N heapsort` | Q6 sorts by price, not review_count. |

What this suggests: review_count-only indexing is relevant to Q3's ordering in these artifacts, but the category/status predicates still remain filters on the index scan.

What not to conclude yet: do not claim workload-wide usefulness from Q3 alone.

## Cross-Index Summary

- The status index was ignored for all Q1-Q6 cases across all three profiles.
- The price index was chosen only for Q6, the price-only control query.
- The created_at index was chosen for Q1 and Q5 across all profiles, and for Q4 on moderate/high skew, but not for Q2, Q3, or Q6.
- The review_count index was chosen only for Q3, the review-count sort query.
- Sort behavior remained visible even when an index was chosen, commonly as `Incremental Sort`.
- Core multi-condition cases Q1 and Q2 were not fully explained by a single-column index in these artifacts.
- Each result file ended with `verify_no_experiment_indexes_remain` returning `(0 rows)`.

This is not a final index conclusion. It is an artifact-supported record of single-column attempt behavior.
